## 0. Objective & scope

Milestone 15b adds a RabbitMQ queue adapter (`RabbitMqQueue`) to the **existing, complete**
`@hono-enterprise/queue-plugin` package from M15. The adapter implements the **committed,
unchanged** internal [`QueueAdapter`](packages/queue-plugin/src/adapters/queue-adapter.ts:20)
transport seam that `MemoryQueue` and `RedisQueue` already implement, and is selected through the
existing `QueuePlugin({ adapter })` option — **no new capability token, no `common` change**.
`common` is not touched in this milestone.

This milestone is the "no-stubs" counterpart to M15 for the RabbitMQ transport, mirroring the M14 →
M14b messaging split: the adapter lands with the full inject-or-lazy `amqplib` client seam and a
guarded real `import('npm:amqplib@0.10.x')` test, exactly like
[`RedisQueue`](packages/queue-plugin/src/adapters/redis-queue.ts:94). The M10 failure mode (an
adapter that echoes input and returns `[]` at 90% coverage) is the explicit anti-goal: the adapter
is driven through a recording fake that asserts the **real AMQP calls** and reads the delivered
payload back.

The committed `QueueAdapter` seam is **polling-based**
([`reserve(name, limit, nowMs)`](packages/queue-plugin/src/adapters/queue-adapter.ts:66) claims due
jobs). RabbitMQ is natively push-based (`consume`), but the
[`QueueService`](packages/queue-plugin/src/services/queue-service.ts:47) owns the poll loop and
calls `reserve`. `RabbitMqQueue` therefore maps the polling seam onto RabbitMQ via **`basicGet`**
(synchronous poll), and uses RabbitMQ's **per-message TTL + dead-letter-exchange** to delay
retry/backoff re-delivery — the "TTL→DLX→requeue" machinery ROADMAP §M15b names. See §3.3 / §3.4 for
the resolution of the "push model" prose in ROADMAP/ARCHITECTURE against the committed polling seam.

- **In scope:**
  - `RabbitMqQueue` — AMQP 0-9-1 via `npm:amqplib@0.10.x`, implementing `QueueAdapter`: per-name
    ready/delay/dead queues on the default exchange; `basicGet`-based `reserve`; per-message TTL +
    DLX for delayed `enqueue`/`requeue`; ack-on-success; publish-to-dead on `deadLetter`; in-memory
    recurring.
  - Extend [`QueueAdapterType`](packages/queue-plugin/src/interfaces/index.ts:81) with `'rabbitmq'`;
    widen [`QueuePluginOptions.client`](packages/queue-plugin/src/interfaces/index.ts:86) to accept
    an AMQP connection; add `prefix?` and the internal `RabbitMqQueueOptions` + structural
    `IAmqpQueueConnection`/`IAmqpQueueChannel`/`IAmqpQueueMessage` (internal, not exported).
  - Extend the plugin's backend selection
    ([`queue-plugin.ts`](packages/queue-plugin/src/plugin/queue-plugin.ts) switch) and the barrel
    ([`src/index.ts`](packages/queue-plugin/src/index.ts)); recording-fake unit test + guarded
    real-import test + fixture; extend `queue-plugin.test.ts`, `barrel-exports.test.ts`,
    `queue-integration.test.ts`.
  - Documentation corrections/additions in `PUBLIC_API.md`, `ARCHITECTURE.md`, `ROADMAP.md` in the
    **same PR** (incl. resolving the staleness/conflicts in §2).
- **NOT this milestone:**
  - Live-RabbitMQ integration tests (a running broker) — deferred; the project bar is
    recording-fake + guarded real-import, matching M15's Redis precedent and M14b (see §9).
  - Priority queues, multiple retry strategies, `removeOnComplete`/`removeOnFail`, visibility
    timeouts / consumer-crash recovery — out of scope of the committed
    [`IQueue`](packages/common/src/services/queue.ts:79) contract (ROADMAP §M15b "NOT in M15b").
  - The `rabbitmq_delayed_message_exchange` plugin, SASL/TLS auth, publisher-confirms — configurable
    later without contract changes.
  - Durable recurring-job storage for the RabbitMQ adapter — recurring metadata is held in-process
    (§3.5); durable recurring is out of scope.
  - Any change to `IQueue`/`QueueAdapter`/`StoredJob`/`StoredRecurring` or a new capability token —
    none (the contract is fixed; the adapter reuses `CAPABILITIES.QUEUE`).

## 3. Design decisions

### 3.1 The inject-or-lazy amqplib client seam (mirroring Redis/messaging)

- **Decision:** `RabbitMqQueue` ships three file-local helpers exactly like
  [`RedisQueue`](packages/queue-plugin/src/adapters/redis-queue.ts:24): `loadAmqplib()` doing
  `await import('npm:amqplib@0.10.x')` and returning the module;
  `validateClient(client): client is IAmqpQueueConnection` checking the **exact** methods the
  adapter calls (`createChannel`, `close`); `resolveClient(url, injected?)` preferring an injected
  client (validated) and otherwise `await amqplib.connect(url)`. `validateClient` is exported from
  the adapter file for direct unit test; `loadAmqplib`/`resolveClient` stay file-local. The
  structural types `IAmqpQueueConnection`/`IAmqpQueueChannel`/`IAmqpQueueMessage` live in
  `interfaces/index.ts` and are **not** barrel-exported (same as `IRedisQueueClient`).
- **Why:** This is the committed M15/M14b pattern and AI_GUIDELINES §12.2; it keeps `amqplib` out of
  the bundle and makes the branching unit-testable through the injection seam. The same
  `npm:amqplib@0.10.x` specifier is already resolved in `deno.lock` from messaging M14b.
- **When the lazy import succeeds / fails:** `loadAmqplib()` resolves when `amqplib` is in the
  workspace cache (it is — M14b); when absent, the dynamic `import('npm:amqplib@0.10.x')` rejects.
  With no injected client and a non-existent endpoint (`amqp://localhost:9999`), `connect()` also
  rejects during the real handshake — so the guarded real-import test (§6) always enters
  `loadAmqplib()` regardless of install.
- **Test home:** `rabbitmq-queue.test.ts` — `validateClient` rejects null/non-object/partial and
  accepts the full shape; "resolveClient prefers injected client"; the one guarded real-import test.

### 3.2 Topology: default-exchange direct-to-queue, per-name ready/delay/dead queues

- **Decision:** For each job name `<n>` the adapter declares three durable queues under a
  configurable prefix (default `he.queue`), all reached via the AMQP **default exchange** (`''`,
  which routes by routing-key = queue name, so no `assertExchange`/`bindQueue` is needed):
  - `he.queue.<n>.ready` — `reserve` does `channel.get` here.
  - `he.queue.<n>.delay` — declared via
    `assertQueue('he.queue.<n>.delay', { durable: true, deadLetterExchange: '', deadLetterRoutingKey: 'he.queue.<n>.ready' })`
    (the **camelCase** amqplib option keys — §1 amqplib row; amqplib maps them to the
    `x-dead-letter-*` `arguments`, and a raw `x-dead-letter-exchange` top-level key would be
    silently dropped). `enqueue` (delayed) and `requeue` publish here with per-message `expiration`;
    when the TTL expires RabbitMQ dead-letters the message to the default exchange with routing-key
    = the ready queue name, moving it to the ready queue.
  - `he.queue.<n>.dead` — `deadLetter` publishes here (final resting place; never re-delivered).
    Queues are asserted lazily and idempotently on first use per name (tracked in a
    `#asserted: Set<string>`).
- **Why:** The default exchange avoids custom exchanges/bindings, minimizing both the adapter and
  the recording fake while preserving real AMQP semantics (DLX routing is configured via
  `assertQueue` args). This is the standard RabbitMQ delayed-message pattern without the non-core
  `rabbitmq_delayed_message_exchange` plugin.
- **Test home:** `rabbitmq-queue.test.ts` — asserts `assertQueue` is called for the delay queue with
  the **camelCase** `deadLetterExchange: ''` / `deadLetterRoutingKey: 'he.queue.<n>.ready'` options
  (and plain `{ durable: true }` for ready/dead); `enqueue`/`requeue` publish to the delay queue;
  `deadLetter` publishes to the dead queue.

### 3.3 `reserve` polls via `basicGet` (resolves the "push model" prose vs the polling seam)

- **Decision:** `RabbitMqQueue.reserve(name, limit, _nowMs)` calls
  `channel.get('he.queue.<name>.ready', { noAck: false })` up to `limit` times, stopping at the
  first **`false`** (amqplib's empty-queue sentinel — `channel_model.js:195` resolves `false`, NOT
  `null`, on `BasicGetEmpty`; §1 amqplib row). Each gotten message's `content` (a `Buffer`) is
  decoded to a string (`msg.content.toString('utf8')`) and `JSON.parse`d into a `StoredJob`, then
  held in an in-memory `#processing: Map<string, Map<string, { message; job }>>` (keyed by `name`
  then `id`) so later `ack`/`requeue`/`deadLetter` can `channel.ack`/re-publish the original AMQP
  message. `nowMs` is accepted for seam conformance but not used to filter — the ready queue only
  ever contains due messages (delayed ones remain in the delay queue until their TTL expires),
  mirroring how [`MemoryQueue.deadLetter`](packages/queue-plugin/src/adapters/memory-queue.ts:144)
  accepts an unused `_nowMs`.
- **Why:** The committed [`QueueAdapter`](packages/queue-plugin/src/adapters/queue-adapter.ts:20)
  seam and the [`QueueService`](packages/queue-plugin/src/services/queue-service.ts:181) poll loop
  are polling-based; `basicGet` is RabbitMQ's faithful non-blocking poll and honors `limit` exactly
  without an internal consume-buffer/backpressure that would fight the service's claim/concurrency
  model. Push `consume` would require buffering and would violate the `limit` contract.
- **Test home:** `rabbitmq-queue.test.ts` — `reserve` asserts `get` is called up to `limit` times
  and stops at the first `false`; claims jobs (a second `reserve` returns nothing until
  re-published); returns the round-tripped `StoredJob` payload decoded from the message `Buffer`;
  returns `[]` when `get` resolves `false`; throws when not connected.

### 3.4 Delayed re-delivery via per-message TTL + DLX (the "TTL→DLX→requeue" cycle)

- **Decision:** every payload is encoded as `Buffer.from(JSON.stringify(job), 'utf8')` before
  `publish` (amqplib rejects a `string`/`Uint8Array` — §1 amqplib row, §3.9); call this `content`
  below.
  - `enqueue(job)`: if `job.availableAtMs <= runtime.now()` (no delay),
    `publish('', 'he.queue.<name>.ready', content)` directly; else
    `publish('', 'he.queue.<name>.delay', content, { expiration: job.availableAtMs - runtime.now() })`
    — the delay queue's DLX routes the expired message to the ready queue.
  - `requeue(name, id, availableAtMs, attempts)`: look up `{ message, job }` in `#processing`; set
    `job.attempts = attempts`, `job.availableAtMs = availableAtMs`;
    `publish('', 'he.queue.<name>.delay', Buffer.from(JSON.stringify(job), 'utf8'), { expiration: Math.max(1, availableAtMs - runtime.now()) })`
    (the "fresh TTL"); `channel.ack(message)` (the original is consumed); remove from `#processing`.
    No-op if not in `#processing` (matches Redis/Memory early-return).
  - `deadLetter(name, id, nowMs)`: look up `{ message, job }`;
    `publish('', 'he.queue.<name>.dead', Buffer.from(JSON.stringify(job), 'utf8'), { timestamp: nowMs })`;
    `channel.ack(message)`; remove from `#processing`. No-op if not in `#processing`.
- **Why:** Retry/backoff is decided by
  [`runJob`](packages/queue-plugin/src/processors/job-processor.ts:59) which calls
  `adapter.requeue(name, id, availableAtMs, attempts)`; the adapter makes the job available at
  `availableAtMs` by handing the delay to RabbitMQ's per-message TTL + DLX. This is the ROADMAP
  "requeue queue ... re-publishes with a fresh TTL", with the adapter's `requeue` performing the
  re-publish directly (no separate requeue-consumer — behaviorally identical and simpler).
  `deadLetter` is the "DLX configuration for failed jobs" — a final dead queue.
- **Test home:** `rabbitmq-queue.test.ts` — `enqueue` (delayed) asserts `publish` to the delay queue
  with the right `expiration`; `requeue` asserts `publish` to the delay queue with
  `expiration = availableAtMs - now` + `ack` of the original; `deadLetter` asserts `publish` to the
  dead queue + `ack`.

### 3.5 Recurring jobs held in-process (mirror MemoryQueue)

- **Decision:** `storeRecurring`/`fetchRecurringDue`/`advanceRecurring` are implemented with an
  in-memory `Map<string, StoredRecurring>` (filter by `nextRunAtMs <= nowMs`; update `nextRunAtMs`
  on advance), exactly as [`MemoryQueue`](packages/queue-plugin/src/adapters/memory-queue.ts:163)
  does. RabbitMQ has no native cron or sorted-set primitive.
- **Why:** The recurring schedule is driven by the
  [`QueueService`](packages/queue-plugin/src/services/queue-service.ts:243) recurring loop, which
  only needs "which recurrings are due?"; an in-process store is co-located with that loop and
  matches the Memory precedent. The actual job delivery still uses the durable RabbitMQ transport.
- **Tradeoff (documented, not blocking):** recurring definitions are lost on process restart and
  must be re-registered via `addRecurring` — same as `MemoryQueue`. Durable recurring for the
  RabbitMQ adapter is out of scope (§9); Redis remains the durable-recurring transport.
- **Test home:** `rabbitmq-queue.test.ts` — `storeRecurring`/`fetchRecurringDue`/`advanceRecurring`
  round-trip and due-filtering; no-op when not found.

### 3.6 `RabbitMqQueue` takes `IRuntimeServices` (justified difference from Redis)

- **Decision:** `new RabbitMqQueue(runtime, options)` — the constructor takes `IRuntimeServices`
  (like messaging's
  [`RabbitMqBroker`](packages/messaging-plugin/src/brokers/rabbitmq-broker.ts:103)), used only to
  convert the absolute `availableAtMs` (carried by `StoredJob`/`requeue`) into a relative
  per-message `expiration`. The plugin passes `ctx.runtime`.
- **Why:** RabbitMQ's TTL is **relative** (ms), while the seam carries **absolute** `availableAtMs`;
  the adapter needs `runtime.now()` to convert.
  [`RedisQueue`](packages/queue-plugin/src/adapters/redis-queue.ts:100) needs no runtime because
  Redis ZSET scores are absolute and `reserve` filters by the passed `nowMs`. No `Date.now()`
  (CLAUDE.md "Never mix clocks").
- **Test home:** `rabbitmq-queue.test.ts` — a delayed `enqueue` and a `requeue` assert
  `expiration = availableAtMs - runtime.now()` using the fake runtime's controlled clock.

### 3.7 Options surface — extend the type, widen `client`, no dead options

- **Decision:**
  - `QueueAdapterType` becomes `'memory'` \| `'redis'` \| `'rabbitmq'`.
  - `QueuePluginOptions.client` is **widened** from `IRedisQueueClient` to `IRedisQueueClient` \|
    `IAmqpQueueConnection`. This widening is non-breaking for _callers_ but it **does** break the
    existing redis branch's forward at
    [`queue-plugin.ts:61`](packages/queue-plugin/src/plugin/queue-plugin.ts) —
    `{ client: options.client }` no longer assigns to `RedisQueueOptions.client`
    (`IRedisQueueClient`), because `options.client` is now the union. **Resolution (required,
    §3.7.1):** in `register()`, narrow the union per branch with each adapter's exported
    `validateClient` type guard before forwarding: the redis branch forwards `options.client` only
    when the redis `validateClient(options.client)` guard passes (narrowing it to
    `IRedisQueueClient`); the rabbitmq branch forwards it only when the amqp `validateClient` guard
    passes (narrowing to `IAmqpQueueConnection`). No cast, no `as`. See §3.7.1 for the exact wiring
    and the one accepted behavior nuance.
  - New flat option `prefix?: string` (default `'he.queue'`) — the queue-name namespace, consumed by
    `RabbitMqQueue`'s queue naming. `url` is reused for the amqp URL (default
    `'amqp://localhost:5672'`).
  - Internal `RabbitMqQueueOptions { url?; client?: IAmqpQueueConnection; prefix? }` (analogous to
    `RedisQueueOptions`) holds exactly the fields the constructor reads; `register()` builds it by
    **assigning only defined values** (mirroring
    [`queue-plugin.ts:61`](packages/queue-plugin/src/plugin/queue-plugin.ts)) to satisfy
    `exactOptionalPropertyTypes`.
- **Why:** Reuses the M15 `url`/`client` pattern; every option names a consumer (no dead surface);
  the internal option type keeps the constructor parallel to `RedisQueue`.
- **Test home:** `queue-plugin.test.ts` extended — drives `adapter: 'rabbitmq'` backend selection
  with an injected fake AMQP connection, asserts options are forwarded (`prefix` reaches queue
  names, `url`/`client` reach `resolveClient`), health + `onClose` wired, unknown adapter still
  throws.

### 3.7.1 `register()` switch — exact wiring for the widened `client` (resolves the type break)

- **Decision:** `queue-plugin.ts` imports both adapters' exported guards under aliases —
  `import { validateClient as isRedisQueueClient } from '../adapters/redis-queue.ts'` and
  `import { validateClient as isAmqpQueueConnection } from '../adapters/rabbitmq-queue.ts'` (they
  share the name `validateClient`, so aliasing is required). Each branch narrows the union before
  forwarding:

  ```ts
  case 'redis': {
    const client = options?.client;
    adapter = new RedisQueue({
      url: options?.url ?? 'redis://localhost:6379',
      ...(client !== undefined && isRedisQueueClient(client) ? { client } : {}),
    });
    break;
  }
  case 'rabbitmq': {
    const client = options?.client;
    adapter = new RabbitMqQueue(ctx.runtime, {
      url: options?.url ?? 'amqp://localhost:5672',
      ...(client !== undefined && isAmqpQueueConnection(client) ? { client } : {}),
      ...(options?.prefix !== undefined ? { prefix: options.prefix } : {}),
    });
    break;
  }
  ```

  The type guard narrows the union to the arm each adapter's `Options.client` requires — no `as`, no
  `@ts-ignore`. The `? {...} : {}` ternary keeps `exactOptionalPropertyTypes` satisfied (never
  assigns `undefined`).
- **Accepted behavior nuance (documented, not blocking):** M15's redis branch forwarded ANY defined
  `client` unconditionally, so a wrong-shaped client reached `resolveClient`, which threw the
  descriptive `Injected Redis client does not match the required structural shape` error. With the
  guard, a client that fails `isRedisQueueClient` under `adapter: 'redis'` is instead dropped and
  the branch lazy-loads ioredis (failing later at connect/import). For a correctly-shaped redis
  client the guard passes and behavior is byte-for-byte identical; only the _misconfiguration_ error
  path changes (a wrong-shaped client is now a connect failure, not a shape-mismatch message). This
  is acceptable and is the price of a single shared `client` field across two adapter shapes.
- **Test home:** `queue-plugin.test.ts` — the existing redis-client-forwarding test still passes
  (correct shape → forwarded); a new test asserts a valid fake AMQP connection under
  `adapter: 'rabbitmq'` is forwarded to `RabbitMqQueue` (reaches `resolveClient`, no lazy import).

### 3.8 Clock, runtime, and lifecycle wiring

- **Decision:** Time comes only from `IRuntimeServices.now()` (TTL computation); no `Date.now()`, no
  global timers. The adapter adds **no** poll loop of its own — the `QueueService` owns the
  worker/recurring loops and calls `reserve`/`fetchRecurringDue`. `connect()` resolves the client,
  opens one channel, and is idempotent; `disconnect()` clears `#processing`, closes the channel, and
  closes the connection only when it was lazy-loaded (not injected); `isReady()` reflects `#ready`.
  Health + lifecycle reuse the plugin's existing `service.createHealthIndicator()` (which delegates
  to `adapter.isReady()`) and `onClose(() => service.disconnect())` unchanged.
- **Why:** CLAUDE.md "Never mix clocks" + AI_GUIDELINES §4; reuses M15 wiring.
- **Test home:** `rabbitmq-queue.test.ts` — `isReady()` false before connect, true after, false
  after disconnect; `connect()` idempotent; `disconnect()` handles a never-connected adapter;
  `enqueue`/`reserve` throw when not connected.

### 3.9 Payload encoding: Node `Buffer` via a static `node:buffer` import (FLAGGED deviation)

- **Decision:** `RabbitMqQueue` encodes every publish payload as a Node `Buffer` —
  `Buffer.from(JSON.stringify(x), 'utf8')` — and decodes on `reserve` with
  `msg.content.toString('utf8')`. `Buffer` is imported statically at the top of `rabbitmq-queue.ts`:
  `import { Buffer } from 'node:buffer';`.
- **Why this is forced, not a preference:** amqplib's frame codec calls `Buffer.isBuffer(content)`
  and throws `TypeError('content is not a buffer')` otherwise (`connection.js:488`, §1 amqplib row).
  A `string` fails it; a web-standard `Uint8Array` **also** fails it (verified —
  `Buffer.isBuffer(new TextEncoder().encode('x')) === false`). There is no web-standard value
  amqplib accepts, so a real `Buffer` is mandatory. (This is exactly why M14b's
  `RabbitMqBroker.publish` — which passes a `Uint8Array` — cannot work against a real broker; §1
  caveat row. M15b does not repeat that.)
- **Deviation flagged (CLAUDE.md "no runtime-specific APIs outside `packages/runtime`"):** this is a
  deliberate, scoped deviation and is called out here rather than slipped in. Justification: (a) the
  requirement comes from the transport library's wire format, with no web-standard substitute; (b)
  `node:buffer` is a **static** `node:` import supported identically by Deno, Node, and Bun — the
  sanctioned form CLAUDE.md endorses for loading Node builtins (never `require`, never
  `globalThis.Buffer`, never a `__x` shim); (c) it passes every gate — AI_GUIDELINES §4.1 scopes the
  hard `node:`/`globalThis` ban to `kernel`/`common`, and no lint rule or CI grep enforces it for
  plugin packages (confirmed: the repo's `deno lint` config is `recommended` + four named rules, no
  node-import gate). The import is confined to `rabbitmq-queue.ts`; `Buffer` never appears in the
  public surface (payloads cross the seam as `StoredJob`, JSON on the wire). A one-line note is
  added to the adapter's module JSDoc explaining the forced `node:buffer` use.
- **Test home:** `rabbitmq-queue.test.ts` + the fake — the fake's `publish` asserts
  `Buffer.isBuffer(content) === true` (so any regression to a `string`/`Uint8Array` payload FAILS
  the test, not silently passes); a round-trip test decodes the recorded `content` back to the
  `StoredJob` and reads its `data`/`attempts`.

## 8. Risks & mitigations

- **Recording fake diverges from the real `amqplib` shape → tests lie.** This is the milestone's top
  risk — three real amqplib facts (verified against the installed `amqplib@0.10.9`, §1) would
  otherwise ship green behind a lenient fake, and this is exactly how M14b's
  `RabbitMqBroker.publish` shipped broken. Mitigations, all mandatory: (a) `publish` content is a
  Node `Buffer` and the fake **asserts `Buffer.isBuffer(content)`** and throws otherwise, mirroring
  `connection.js:488` (§3.9); (b) `get` returns **`false`** (not `null`) on an empty queue and the
  adapter/fake both honor the `BasicGetEmpty` sentinel (§3.3); (c) the delay queue is asserted with
  the **camelCase** `deadLetterExchange`/`deadLetterRoutingKey` options amqplib actually reads
  (§3.2) — a raw `x-...` key would silently drop the DLX. The fake mirrors real method names/returns
  (`assertQueue`→`{queue}`, `publish`→`boolean`, `get`→`Promise<Msg|false>`, `ack`→void), and the
  one guarded real-import test exercises the real `import('npm:amqplib@0.10.x')` path so the load
  function is never only covered by the fake.
- **`node:buffer` import is a runtime-specific-API deviation.** Mitigation: forced by amqplib's wire
  format (no web-standard value passes `Buffer.isBuffer`), scoped to `rabbitmq-queue.ts`, uses the
  sanctioned static `node:` import form, passes all gates, and is flagged as a design decision
  (§3.9) with a module-JSDoc note — not slipped in.
- **Widening `QueuePluginOptions.client` breaks the existing redis-branch forward.** Mitigation:
  `register()` narrows the union per branch via each adapter's exported `validateClient` type guard
  (§3.7.1) — no cast; the one behavior nuance (a wrong-shaped client under `adapter:'redis'` now
  fails at connect rather than with the shape-mismatch message) is documented and the existing
  redis-forward test still passes.
- **Per-message TTL head-of-line blocking in the delay queue.** Mitigation: a known RabbitMQ
  limitation (a long-TTL message ahead of a short-TTL one delays the short one); documented; for
  production the `rabbitmq_delayed_message_exchange` plugin is the mitigation (out of scope, §9).
  The unit test asserts the `expiration` value; the integration fake routes delay→ready immediately
  to exercise the re-delivery mechanism without depending on real TTL timing.
- **Recurring metadata is non-durable for the RabbitMQ adapter.** Mitigation: documented (§3.5);
  matches `MemoryQueue`; Redis remains the durable-recurring transport; durable recurring for
  RabbitMQ is explicitly out of scope.
- **Coverage of the lazy-load line is environment-gated.** Mitigation: the guarded real-import test
  enters `loadAmqplib()` with no injected client so the function is covered whether or not `amqplib`
  is installed (redis precedent at
  [`redis-queue.test.ts:497`](packages/queue-plugin/test/unit/redis-queue.test.ts)).
- **`exactOptionalPropertyTypes` is on.** Mitigation: the `RabbitMqQueueOptions` object is built by
  assigning only defined values (mirroring
  [`queue-plugin.ts:61`](packages/queue-plugin/src/plugin/queue-plugin.ts)).
- **`noUnusedParameters` flags the unused `nowMs` in `reserve`.** Mitigation:
  `reserve(name, limit, _nowMs)` follows the sanctioned param convention already used by
  [`MemoryQueue.deadLetter(name, id, _nowMs)`](packages/queue-plugin/src/adapters/memory-queue.ts:144);
  the param is required by the `QueueAdapter` seam signature.

## 9. Out of scope

- Live-RabbitMQ integration tests (a running broker) — deferred; the project bar for transport
  adapters is recording-fake + guarded real-import (M15's Redis precedent, M14b), not live backends.
- Priority queues, multiple retry strategies, `removeOnComplete`/`removeOnFail`, visibility timeouts
  / consumer-crash recovery — out of scope of the committed
  [`IQueue`](packages/common/src/services/queue.ts:79) contract (ROADMAP §M15b "NOT in M15b").
- The `rabbitmq_delayed_message_exchange` plugin, publisher-confirms, SASL/TLS auth — configurable
  later without contract changes.
- Durable recurring-job storage for the RabbitMQ adapter (§3.5) — recurring metadata is in-process;
  durable recurring is out of scope.
- Any modification to `IQueue`/`QueueAdapter`/`StoredJob`/`StoredRecurring` or a new capability
  token — none (owned by `common`/the M15 seam, intentionally untouched).
