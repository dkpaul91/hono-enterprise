# Milestone 24 — Telemetry Plugin (`@hono-enterprise/telemetry-plugin`, `@hono-enterprise/common`)

> **Status:** Planning. Branch: `feat/24-telemetry-plugin`. `main` is protected — all work
> (implementation + fixes) stays on this one branch until it merges via a single PR.

## 0. Objective & scope

Provide **OpenTelemetry distributed tracing** as a plugin. The `TelemetryPlugin` registers an
`ITelemetryService` under `CAPABILITIES.TELEMETRY` (`'telemetry'`), exposing manual span creation
(`withSpan`) plus a request-span middleware that wraps every inbound HTTP request in a server span
with W3C `traceparent`/`tracestate` propagation. The OpenTelemetry SDK is a **heavy optional
dependency** (AI_GUIDELINES §12.2): the plugin lazy-loads `@opentelemetry/sdk-trace-base` and an
exporter (`@opentelemetry/exporter-trace-otlp-http` or the in-process console exporter) via dynamic
`npm:` imports, failing with a clear error when the package is absent; an injectable
`TracerProviderFactory` seam lets tests (and consumers with a pre-built provider) bypass the lazy
import entirely. A `NoopTelemetryService` is the default when no exporter is configured, so the
plugin always registers a usable service with zero npm deps in that mode.

This milestone **adds the telemetry contract to `common`** — `ITelemetryService`, `ISpan`,
`SpanStatus`, `SpanKind`, and `TelemetryPluginOptions`-adjacent types — because `common` currently
ships only the `CAPABILITIES.TELEMETRY` token (`packages/common/src/tokens.ts:71`) and no service
interface. That is a public-API change: `PUBLIC_API.md` and `ARCHITECTURE.md` are updated in the
same PR.

- **In scope:**
  - New contract in `packages/common/src/services/telemetry.ts`: `ITelemetryService`, `ISpan`,
    `SpanStatus`, `SpanKind`, `SpanOptions`, `TelemetryContext`. Exported from
    `packages/common/src/index.ts`.
  - `TelemetryPlugin` factory in `packages/telemetry-plugin/src/plugin/telemetry-plugin.ts`
    registering `ITelemetryService` under `CAPABILITIES.TELEMETRY`.
  - `TelemetryService` (`src/services/telemetry-service.ts`) wrapping an OTel
    `BasicTracerProvider`/`Tracer` when an exporter is configured, or a zero-dep
    `NoopTelemetryService` when not.
  - `src/tracing/tracer.ts` — internal `TracerHost` seam + the lazy OTel loader
    (`loadOtelTracerProvider`).
  - `src/exporters/otlp-exporter.ts` — lazy `@opentelemetry/exporter-trace-otlp-http` loader.
  - `src/exporters/console-exporter.ts` — `ConsoleSpanExporter` loader (from
    `@opentelemetry/sdk-trace-base`).
  - `src/middleware/telemetry-middleware.ts` — request-span middleware (priority 30, outermost after
    metrics at 20 so the trace observes the full request including metrics recording), propagating
    `traceparent`/`tracestate` in/out and storing the active span on `ctx.state` under a documented
    key.
  - `src/index.ts` barrel exports.
  - `PUBLIC_API.md` (Telemetry section expanded with the real contract), `ARCHITECTURE.md`
    (telemetry-plugin row already present; add the contract + middleware-priority note),
    `ROADMAP.md` (file list + Deliverables/Tests lists corrected per §2 C4/C5 — only what M24
    actually ships gets checked — the progress row flipped, and the new M24b section + `24b`
    progress row created per §2 C6).
- **NOT this milestone:**
  - **Auto-instrumentation of fetch/HTTP clients, database drivers, message brokers** — the ROADMAP
    lists `instrumentations: ['http', 'database', 'queue']` as a registration option, but
    `@opentelemetry/*-instrumentation` packages are runtime-specific (Node `http`, fetch, etc.) and
    pulling them in would violate runtime independence (AI_GUIDELINES §4) and the §12.2 heavy-dep
    rule. M24 ships **manual spans + the request-span middleware only**; the `instrumentations`
    option is **cut** (see §3.4 — a planned option no implementation can honestly consume across
    runtimes is a dead option). Auto-instrumentation is owned by a follow-up **M24b** that adds
    runtime-gated instrumentation packages behind the same inject-or-lazy seam — **§9.1 is the
    authoritative M24b scope**, and §2 C6 makes creating M24b's ROADMAP entry a deliverable of THIS
    PR (a `deferred to M24b` note pointing at a milestone that exists nowhere is a dangling
    reference).
  - **Metrics/Logs signals** (OTel metrics SDK, logs SDK) — owned by the existing `metrics-plugin`
    (Prometheus) and `logger-plugin`; OTel-SDK-based metrics/logs bridges are a later concern.
  - **Context propagation across async boundaries without an OTel context manager** — M24 uses the
    OTel `context` module when the SDK is loaded; the `NoopTelemetryService` does not propagate
    across `await` boundaries (it is request-scoped via `ctx.state`). Cross-process propagation is
    limited to `traceparent` header injection/extraction in the middleware.

## 3. Design decisions

### 3.1 Contract lives in `common` — framework-owned types, not OTel re-exports

- **Decision:** `packages/common/src/services/telemetry.ts` defines `ITelemetryService`, `ISpan`,
  `SpanStatus` (`'ok' | 'error' | 'unset'`), `SpanKind`
  (`'internal' | 'server' | 'client' | 'producer' | 'consumer'`), `SpanOptions`
  (`{ kind?: SpanKind; attributes?: Readonly<Record<string, SpanAttributeValue>>; parentSpan?: ISpan }`),
  and `SpanAttributeValue` (`string | number | boolean | ReadonlyArray<string | number | boolean>`).
  These are **framework-owned unions/interfaces**, NOT re-exports of `@opentelemetry/api` enums
  (which would make `common` depend on an npm package — forbidden: `common` has zero deps,
  `packages/common/deno.json` and AI_GUIDELINES §2.1). The `TelemetryService` implementation
  translates framework `SpanStatus`/`SpanKind` to OTel's `SpanStatusCode`/`SpanKind` at the seam.
- **Why:** Every other service contract in `common` (`ILogger`, `IMetricsService`, `IQueue`, etc.)
  is framework-owned and dependency-free; telemetry must follow the same pattern. Re-exporting OTel
  types would either add a dep to `common` or require `import type` from an npm specifier that may
  not be installed (breaking `deno check` for consumers who do not use telemetry).
- **Test home:** `packages/common/test/unit/telemetry.test.ts` (NEW) — type-level compile test (the
  contract is interface-only; `deno check` is the gate, plus a structural test that a fake
  `ITelemetryService` satisfies the interface).

### 3.2 `withSpan` — the single manual-span API

- **Decision:**
  `ITelemetryService.withSpan<T>(name: string, fn: (span: ISpan) => Promise<T>, options?: SpanOptions): Promise<T>`
  is the only manual span-creation method. `ISpan` exposes `setAttribute(name, value)`,
  `setAttributes(attrs)`, `setStatus(status)`, `recordException(error)`, `end()`, `setAttribute`
  returns `this` for chaining. The callback's return value is returned by `withSpan`; the span is
  ended in a `finally` block. There is no separate `startSpan`/`endSpan` pair on the public
  interface — `withSpan` is the ergonomic, leak-proof surface (a `startSpan` that the caller forgets
  to `end()` is a real OTel leak).
- **Why:** The PUBLIC_API.md example (`telemetry.withSpan('create-order', async (span) => {...})`)
  already shows this shape; making it the only API prevents the "forgot to end" class of bug and
  matches the framework's composition-over-inheritance rule. An internal `startSpan` exists on the
  `TracerHost` seam (used by the middleware) but is NOT exported.
- **Test home:** `test/unit/telemetry-service.test.ts` — `withSpan` returns the callback's value;
  `span.end()` is called exactly once even when the callback throws; `recordException` records the
  error and sets status to `'error'`; attributes set before `end()` are visible to the exporter.

### 3.3 Noop default + lazy OTel — the inject-or-lazy `TracerHost` seam

- **Decision:** `TelemetryPlugin({ serviceName })` with no `exporter` option registers a
  `NoopTelemetryService` (zero deps, `withSpan` just runs the callback with a `NoopSpan` whose
  setters are no-ops; `end()` is a no-op). When `exporter: 'otlp'` (with `endpoint`) or
  `exporter: 'console'` is set, the plugin's `register()` (which MAY return a `Promise`) calls
  `loadOtelTracerProvider({ serviceName, serviceVersion, exporter, endpoint, sampling })` which
  lazy-loads `npm:@opentelemetry/sdk-trace-base@^2.9.0`, `npm:@opentelemetry/resources@^2.9.0`, and
  (for OTLP) `npm:@opentelemetry/exporter-trace-otlp-http@^0.220.0`, then builds the provider with
  the **2.x constructor-config** shape — `addSpanProcessor()` does not exist on this line (§1):

  ```ts
  new BasicTracerProvider({
    resource: resourceFromAttributes({
      'service.name': serviceName,
      'service.version': serviceVersion,
    }),
    spanProcessors: [new SimpleSpanProcessor(exporterInstance)],
    sampler: new TraceIdRatioBasedSampler(ratio),
  });
  ```

  and returns a `TracerHost` wrapping its `getTracer(serviceName)` plus `shutdown`/`forceFlush`. The
  class is `BasicTracerProvider` — there is no `TracerProvider` export, and `NodeTracerProvider`
  (Node-only, in `sdk-trace-node`) is deliberately not used. An injectable
  `tracerProviderFactory?: () => Promise<TracerHost>` option lets a consumer (or test) supply a
  pre-built provider, bypassing the lazy import — identical to the `RedisRateLimitStore({ client })`
  injection seam.
- **Why:** AI_GUIDELINES §12.2 forbids heavy deps as hard dependencies; the OTel SDK is heavy and
  Node-leaning (`NodeTracerProvider` pulls `node:http`). `BasicTracerProvider` +
  `SimpleSpanProcessor` from `sdk-trace-base` is runtime-agnostic. The noop default means
  `TelemetryPlugin()` is always safe to register (matches the "secure defaults" / "always usable"
  pattern of metrics/health plugins). `register()` returning a `Promise` is sanctioned (CLAUDE.md
  "Common pitfalls": "If a real `import()` forces the construction path to be async, make it
  async").
- **Test home:** `test/unit/telemetry-plugin.test.ts` — noop mode registers a service with no
  `await import`; `test/unit/tracer.test.ts` — `loadOtelTracerProvider` calls the lazy import with
  the pinned specifier (driven by a fake `TracerProviderFactory`);
  `test/integration/otlp-exporter.test.ts` — guarded REAL
  `await import('npm:@opentelemetry/exporter-trace-otlp-http@^0.220.0')` skipped when the dep is
  absent, exercising the real load path once.

### 3.4 `instrumentations` option — CUT (dead option)

- **Decision:** The `instrumentations: ['http', 'database', 'queue']` option shown in PUBLIC_API.md
  and ROADMAP.md is **not implemented** and **fully removed** from `TelemetryPluginOptions` — the
  type carries **no** `instrumentations` field, so passing it is a **compile-time error**
  (`deno
  check`), not a silent no-op. (An early implementation kept it as an accepted-but-ignored
  `@deprecated` field; that was removed — a field no code reads is dead surface, and a silent no-op
  is a worse failure mode than a loud compile error. There is deliberately **no placeholder**, so
  M24b defines the real option free of any published-shape back-compat constraint.)
- **Why:** `@opentelemetry/instrumentation-http` instruments Node's `http` module; there is no
  Deno/Bun/CF-Workers equivalent in the OTel ecosystem that this plugin could load without a runtime
  branch (which would violate §4). An option the plugin cannot honestly consume on every runtime is
  a dead option (CLAUDE.md "every option names its consumer"). Cutting it — from the type, not just
  the behavior — is the prescribed fix, not "store it and ignore it". The real option is owned by
  M24b (ROADMAP §Milestone 24b, deliverable "Public `instrumentations` option"), which defines a
  per-instrumentation shape, not a bare `string[]`.
- **Test home:** `test/unit/telemetry-plugin.test.ts` — the absence of `instrumentations` from the
  `TelemetryPluginOptions` type is a `deno check`-level guarantee (no runtime test asserts a no-op,
  because there is no accepted option to exercise).

### 3.5 Request-span middleware — priority 30, `traceparent` propagation, span on `ctx.state`

- **Decision:** `telemetryMiddleware()` is registered at priority **30** (inside metrics at 20, so
  the request span is the parent of the metrics observation; outside auth at 300 so the trace
  captures auth failures). It: (1) extracts `traceparent`/`tracestate` from `ctx.request.headers`
  (W3C Trace Context), (2) starts a server span named `<METHOD> <path>` via the internal
  `TracerHost.startSpan` with `kind: 'server'` and the extracted parent context, (3) stores the span
  on `ctx.state` under the exported constant key `TELEMETRY_SPAN_KEY = '__he_telemetry_span'`, (4)
  sets HTTP attributes (`http.method`, `http.route`, `http.status_code` after `next()`), (5) in a
  `finally` ends the span and, if the response is available, injects the span's `traceparent` back
  into `ctx.response` via `ctx.response.header('traceparent', ...)`. On a noop service, the
  middleware still runs (sets a noop span on `ctx.state`) so downstream code reading
  `ctx.state.get(TELEMETRY_SPAN_KEY)` gets a usable `ISpan`.
- **Why:** Priority 30 places the trace span outside auth/validation but inside metrics — the
  standard OTel "server span wraps the whole inbound request" pattern. Storing the span on
  `ctx.state` (not a new `IRequestContext` field) avoids a `common` contract change and uses the
  sanctioned per-request bag. `traceparent` injection on the response is optional and only happens
  when the service is real (noop skips it).
- **Test home:** `test/unit/telemetry-middleware.test.ts` — span started with `kind: 'server'`;
  `traceparent` request header is extracted as the parent; span on `ctx.state` under the key;
  `http.status_code` attribute set from the response snapshot after `next()`; span ended in
  `finally` even when `next()` throws; `traceparent` response header set (real service) / absent
  (noop); short-circuit by a downstream middleware (no `next()`) still ends the span.

### 3.6 Sampling — `traceidratio` only, mapped to OTel's `TraceIdRatioBasedSampler`

- **Decision:** `TelemetryPluginOptions.sampling?: { type: 'traceidratio'; ratio: number }` is the
  only sampling config. `type` is a literal-union with one member now (extensible later without a
  breaking change). The lazy loader maps it to OTel's `TraceIdRatioBasedSampler(ratio)`, passed as
  the `sampler` field of the `BasicTracerProvider` **constructor config** (§3.3 — there is no
  post-construction setter on the 2.x line); default ratio `1.0` (sample everything). `alwaysOn` is
  the noop-mode behavior (no sampler needed).
- **Why:** `parentbased`/`alwaysOff` add surface with no current consumer; `traceidratio` is the one
  sampling strategy the PUBLIC_API.md example shows. A single-member union is forward-compatible.
- **Test home:** `test/unit/tracer.test.ts` — `loadOtelTracerProvider` passes the sampler config to
  the provider (driven by a fake factory that records the config).

### 3.7 Shutdown — flush the provider on `onShutdown`

- **Decision:** The plugin registers `ctx.lifecycle.onShutdown(() => provider.shutdown())` when a
  real provider exists. `NoopTelemetryService` has no shutdown. `ITelemetryService` does NOT expose
  `shutdown` (it is a plugin-lifecycle concern, not a service-contract method — matching
  `IMetricsService` which also has no `shutdown`).
- **Why:** `BatchSpanProcessor` buffers spans; `shutdown()` flushes them. Tying it to `onShutdown`
  (not the service interface) keeps the contract clean and matches the kernel's graceful-shutdown
  model.
- **Test home:** `test/unit/telemetry-plugin.test.ts` — `onShutdown` hook calls
  `provider.shutdown()` (fake provider records the call); noop mode registers no `onShutdown` hook.

## 8. Risks & mitigations

- **OTel's two independent version lines — the trap this plan already fell into once.** The stable
  packages (`sdk-trace-base`, `resources`, `core`, `api`) are on **2.x** (`sdk-trace-base@2.9.0`),
  while the experimental OTLP exporters are on **0.2xx.x** (`exporter-trace-otlp-http@0.220.0`).
  Assuming one line covers both yields `npm:@opentelemetry/sdk-trace-base@^0.220.0`, a specifier
  that **resolves to nothing** — the lazy import fails at runtime in exactly the mode where the
  plugin is supposed to work, while every gate stays green (the failure lives behind a lazy import
  no unit test executes). Mitigation: §1 records both lines from the real registry + a real import
  probe; `test/unit/tracer.test.ts` asserts the exact specifier strings; the §7 grep re-checks them;
  and the guarded integration test executes the real import once.
- **OTel 2.x removed `addSpanProcessor()`.** The pre-2.x idiom
  `provider.addSpanProcessor(new SimpleSpanProcessor(exp))` is what most tutorials and most model
  priors still show, but `BasicTracerProvider.prototype.addSpanProcessor` is `undefined` on the
  pinned line (§1, verified). Calling it throws `TypeError: not a function` at runtime. Mitigation:
  §3.3 spells out the constructor-config form; the §7 grep for `addSpanProcessor` must be empty.
- **`NodeTracerProvider` temptation:** OTel's `NodeTracerProvider` (in `sdk-trace-node`) pulls
  `node:http` and is Node-only. M24 deliberately uses `BasicTracerProvider` from `sdk-trace-base`
  (runtime-agnostic) + `SimpleSpanProcessor`. Note `sdk-trace-base` exports **no** `TracerProvider`
  symbol at all (§1) — writing `new TracerProvider(...)` is an undefined-is-not-a- constructor
  error. Mitigation: a reviewer grep for `sdk-trace-node|NodeTracerProvider` in
  `packages/telemetry-plugin` must be empty (§7).
- **No implicit parent/child linking — parents must be passed explicitly (real mode too).** This
  framework registers **no OTel `ContextManager`** (the only runtime-agnostic path to one,
  `AsyncLocalStorageContextManager`, pulls `node:async_hooks` and would violate §4 runtime
  independence — verified 2026-07-21 by import probe). Consequently `api.context.active()` is always
  `ROOT_CONTEXT`, so a `withSpan` nested inside another `withSpan` does **not** auto-link — in noop
  mode AND in real mode alike. Parenting works only when a parent is passed explicitly: OTel's
  `Tracer.startSpan(name, options, context)` reads the parent from the 3rd `context` arg (via
  `api.trace.getSpan(context)`), NOT from anything in `options`, so `loadOtelTracerProvider`'s
  `startSpan` builds that context from `SpanOptions.parentContext` and passes it as the 3rd arg.
  Mitigation: the request-span middleware always passes the extracted remote parent explicitly, so
  the one propagation path the milestone ships (incoming `traceparent` → server span) is correctly
  parented — verified end-to-end against the real OTel SDK (server span inherits the incoming
  traceId + parentSpanId). Application code that wants a child span must pass `parentSpan` /
  `parentContext` on `SpanOptions`; this is documented on `ITelemetryService.withSpan`.
- **Coverage of the lazy-import error path:** the `loadOtelTracerProvider` rejection branch (package
  not installed) must be unit-covered. Mitigation: `test/unit/tracer.test.ts` drives it with a fake
  `tracerProviderFactory` whose `import()` rejects, asserting the clear error message — no guarded
  skip needed for the branching logic.
- **`common` contract addition is a public-API change:** adding `ITelemetryService` to `common` and
  exporting it from `src/index.ts` requires a PUBLIC_API.md update in the same PR (AI_GUIDELINES
  §10.2). Mitigation: §2 C2 + the §4.1 table name the exact doc deliverable.
- **Middleware priority collision:** priority 30 is currently unused (metrics 20, logging 50).
  Mitigation: the §10 middleware table in ARCHITECTURE.md is updated to show the telemetry row at
  30; the test asserts the priority passed to `ctx.middleware.add`.

## 9. Out of scope

### 9.1 Deferred to M24b — the authoritative scope

**M24b is `telemetry-plugin` ONLY**, mirroring the single-package M14→M14b, M15→M15b, and M16→M16b
splits (CLAUDE.md: "Work on **one package per milestone**"). Its entire scope is:

1. **Auto-instrumentation** (`@opentelemetry/instrumentation-http`, fetch, ioredis, amqplib,
   kafkajs) — runtime-gated instrumentation packages loaded behind the same inject-or-lazy
   `TracerHost` seam M24 establishes. **M24b also owns defining the public option that enables it:**
   M24 removed `instrumentations` from the type entirely (§3.4) and ships no placeholder, so M24b
   adds a NEW `TelemetryPluginOptions.instrumentations` field with a **per-instrumentation shape
   (not a bare `string[]`)** — a public-API change (PUBLIC_API.md + the type). Unsupported-runtime
   targets degrade to a documented no-op, never a throw (§4). See ROADMAP §Milestone 24b for the
   full deliverable list.
2. **`BatchSpanProcessor` with flush-on-interval** — **decided: in M24b, not "a follow-up".** M24
   uses `SimpleSpanProcessor` (synchronous export per span) for deterministic test behavior; M24b
   adds `BatchSpanProcessor` as a `TelemetryPluginOptions.spanProcessor` choice, since
   auto-instrumentation raises span volume enough to make per-span synchronous export the wrong
   default. Both processors are already exported from the pinned `sdk-trace-base@^2.9.0` (§1), so
   this adds no new dependency.

Nothing else is M24b. In particular, see §9.2 — cross-package work is NOT M24b.

### 9.2 Out of scope for BOTH M24 and M24b

- **Cross-process propagation over the message broker / queue** — injecting/extracting `traceparent`
  into broker and queue messages requires editing `messaging-plugin` and `queue-plugin`, which M24b
  does not own. A single milestone touching telemetry + messaging + queue would violate the
  one-package-per-milestone rule, so this is **not M24b**: it belongs to a later cross-cutting
  milestone (or to those plugins' own follow-ups), scheduled separately. M24 and M24b limit
  propagation to the inbound/outbound HTTP `traceparent` header handled by `telemetryMiddleware`.
- **OTel Metrics/Logs SDK bridges** — the existing `metrics-plugin` (Prometheus) and `logger-plugin`
  own those signals; an OTel-SDK-based metrics/logs exporter bridge is a later concern.
- **`tracestate` header round-tripping beyond extraction** — M24 extracts `tracestate` for the
  parent context but does not mutate it; full `tracestate` vendor management is out of scope.
